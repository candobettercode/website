01

Introduction to Git

What is Git?

Git is a distributed version control system (DVCS) created by Linus Torvalds in 2005. It tracks changes in your source code, lets you collaborate with others, and allows you to revert to any previous state of your project.

Think of Git as a time machine for your codebase — every commit is a snapshot you can travel back to at any point.

Why Git?

  • Track every change with who made it and why
  • Branch and experiment without affecting main code
  • Collaborate with teams across the globe
  • Merge code from multiple developers automatically
  • Roll back to any previous version instantly
  • Industry standard — used by virtually all software teams

02

Git Basics

Every Git project starts with initialization. Once initialized, your folder becomes a Git repository with a hidden .git directory that tracks all history.

bash — Initialize a repository
# Initialize a new Git repository git init # Initialize in a specific folder git init my-project
bash — Stage & Commit
# Stage all changes git add . # Stage a specific file git add index.html # Commit staged changes with a message git commit -m "feat: add landing page" # Stage + commit in one step (tracked files only) git commit -am "fix: correct typo in header"
bash — Status, Log & Diff
# See working tree status git status # View commit history git log # Compact one-line log with graph git log --oneline --graph --all # See what changed (unstaged) git diff # See what changed (staged) git diff --staged
💡

Tip: Write commit messages in the imperative mood: "add feature" not "added feature". Follow Conventional Commits for team projects.

03

Git Branching

What is a Branch?

A branch is a lightweight, movable pointer to a specific commit. Creating a branch lets you diverge from the main line of development and work independently without affecting other branches.

The default branch is typically called main (or master in older repos). When you create a feature branch, you're making a parallel timeline.

bash — Branch commands
# List all branches git branch # Create a new branch git branch feature/login # Create and switch (classic) git checkout -b feature/login # Create and switch (modern, Git 2.23+) git switch -c feature/login # Delete a merged branch git branch -d feature/login
📌

HEAD is a special pointer that tells Git "where you currently are". It normally points to the tip of your current branch. When you switch branches, HEAD moves with you.

04

Merge vs Rebase

Both merge and rebase integrate changes from one branch into another — but they do it very differently. Choosing the right one shapes your project history.

git merge non-destructive

A─B─C─────F ╲ ╱ D─E

Creates a new merge commit (F) tying both histories together. Preserves full context.

git rebase linear history

A─B─C─D'─E'

Replays feature commits on top of main. Clean linear history but rewrites commit SHAs.

bash — Merge & Rebase
# --- MERGE --- git checkout main git merge feature/login # Fast-forward only git merge --ff-only feature/login # --- REBASE --- git checkout feature/login git rebase main # ⚠️ Never rebase commits already pushed to shared branches!
AspectMergeRebase
HistoryNon-linear (preserves branches)Linear (rewrites)
Commit SHAsUnchangedNew SHAs created
Safe for public branches✅ Yes⚠️ No
Conflict resolutionOnce at mergePer commit during rebase

GitHub

What is GitHub?

GitHub is a cloud-based hosting platform for Git repositories. It adds a collaboration layer on top of Git — pull requests, issues, code review, CI/CD pipelines, and project management tools.

GitHub is where the open-source world lives. Over 100 million developers host their projects there.

Conflict Resolution

A merge conflict occurs when two branches modify the same lines of a file. Git marks the conflict and asks you to resolve it manually.

diff — Conflict markers in file
<<<<<<< HEAD (current branch) const greeting = "Hello World"; ======= const greeting = "Hi Universe"; >>>>>>> feature/update-greeting # After resolving manually, stage and commit: git add . git commit -m "fix: resolve greeting conflict"
07

Advanced Git

bash — Interactive Rebase
# Rewrite last 3 commits interactively git rebase -i HEAD~3 # In the editor, use these actions: # pick → keep commit as-is # reword → change commit message # squash → combine with previous commit # fixup → squash, discard message # drop → remove commit entirely
bash — Git Stash
# Temporarily shelve changes git stash # Stash with a description git stash push -m "WIP: half-done login form" # List stashes git stash list # Apply most recent stash (keep in list) git stash apply # Apply and remove from list git stash pop
bash — Cherry-pick
# Apply a specific commit from another branch git cherry-pick a3f5c2d # Cherry-pick a range of commits git cherry-pick a3f5c2d..e9b12f4
bash — Reset vs Revert
# --- RESET (rewrites history) --- git reset --soft HEAD~1 # undo commit, keep staged git reset --mixed HEAD~1 # undo commit, unstage (default) git reset --hard HEAD~1 # undo commit, discard all changes # --- REVERT (safe for shared branches) --- git revert a3f5c2d # new commit that undoes the specified one
CommandRewrites HistorySafe for RemoteUse Case
reset --softYes⚠️ Local onlyRedo last commit message
reset --hardYes❌ DangerousNuclear undo (local only)
revertNo✅ YesUndo pushed commits safely
bash — Git Reflog (safety net)
# View all recent HEAD movements git reflog # Recover a "lost" commit git checkout HEAD@{3} # Restore branch to a previous state git reset --hard HEAD@{2}
🛟

Reflog is your safety net. Even after a reset --hard, Git keeps commits for ~90 days. Reflog lets you recover them.

08

Git Internals

Under the hood, Git stores everything as objects in the .git/objects directory. Each object is identified by a SHA-1 hash of its content.

Object Type 01

Blob

Stores the raw contents of a file. Blobs don't have filenames — just file data. Two identical files share one blob.

Object Type 02

Tree

A directory listing. Points to blobs (files) and other trees (subdirs), along with their filenames and permissions.

Object Type 03

Commit

Points to a tree (snapshot), parent commit(s), author, committer, timestamp, and message. Commits chain history together.

Object ID

SHA-1 Hash

Every Git object gets a 40-character hexadecimal SHA-1 hash. Content-addressed: same content always produces same hash.

bash — Inspect Git objects
# View object type git cat-file -t a3f5c2d # → commit git cat-file -p a3f5c2d # → show content # View the tree a commit points to git ls-tree HEAD # Get the SHA-1 hash of a file git hash-object README.md

09

Interview Questions

Common Git & GitHub questions asked in technical interviews, from junior to senior level. Click any question to expand the answer.

git fetch downloads changes from the remote but does NOT integrate them into your working branch. git pull is essentially fetch + merge — it downloads and immediately merges the changes. Use fetch when you want to inspect changes before integrating.
git reset moves the branch pointer backward, effectively removing commits from history. It's destructive and should not be used on shared branches. git revert creates a new commit that undoes the changes of a specific commit without rewriting history — safe for public branches.
Normally HEAD points to a branch name (e.g., main). A detached HEAD occurs when HEAD points directly to a commit SHA instead of a branch — this happens when you git checkout <commit-hash>. Commits made in this state aren't associated with any branch. To save work, create a branch: git switch -c new-branch.
git cherry-pick <commit> applies the changes of a specific commit to your current branch. It's useful when you need a specific bug fix from a feature branch applied to main before the full feature is ready, or when you accidentally committed to the wrong branch.
Gitflow: Uses main, develop, feature/*, release/*, hotfix/*. Good for versioned releases.

GitHub Flow: Single main + short-lived feature branches. Simple, works well with CI/CD.

Trunk-Based Development: All developers commit directly to main frequently. Feature flags replace long-lived branches. Preferred by high-velocity teams.
Squashing combines multiple commits into a single one for a cleaner history. Use git rebase -i HEAD~N (change pick to squash), or git merge --squash feature-branch. It's commonly done before merging a feature PR to keep main's history readable.
.gitignore tells Git which files or directories to ignore. Common entries: node_modules/, .env, build artifacts, IDE config. Patterns support wildcards (*.log), negations (!important.log), and directories (dist/). Note: files already tracked by Git won't be ignored just by adding them to .gitignore.
git stash temporarily shelves uncommitted changes so you can switch context — for example, to fix an urgent bug on another branch — without committing unfinished work. The stash is a stack. Use git stash pop to reapply and remove the most recent stash, or git stash apply stash@{N} for a specific one.
